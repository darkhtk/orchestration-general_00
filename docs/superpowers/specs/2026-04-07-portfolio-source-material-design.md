**Date:** 2026-04-07
**Topic:** `Orchestration_general` 케이스 스터디 소스 자료 (이력서·포트폴리오 양쪽에서 발췌해 쓰는 단일 소스 문서)
**Author:** 홍대기 (with Claude Code)

## 1. 목적과 사용처

이 문서는 `Orchestration_general` 프레임워크와 그 검증 결과물인 `Gen` (GenWorld Unity RPG)을 **이력서와 포트폴리오에서 자유롭게 발췌해 쓸 수 있는 단일 소스(case study source-of-truth)** 한 벌을 만드는 작업의 설계 명세다.

산출물은 마크다운 1개 파일이며, 다음 조건을 만족해야 한다.

- **타겟 직무:** AI 엔지니어 / LLM 애플리케이션 개발자(B), 테크 리드 / 시니어 풀스택(D)
- **타겟 시장:** 한국, 한국어 본문
- **사용자의 기존 자산:** 이미 PDF 포트폴리오와 이력서가 있음. 새 디자인을 만들지 않고, 이 소스 문서에서 텍스트와 캡션을 잘라 기존 포맷에 배치한다.
- **퍼블릭 레포 활용:** `Orchestration_general`과 `Gen` 두 레포 모두 GitHub public이므로 코드 전문을 옮기지 않고 링크 + 캡션 큐레이션 방식을 쓴다.

## 2. 핵심 내러티브 형식

**4박자 스토리 구조 (이중 레이어)**

### Layer 1 — 문서 전체가 하나의 큰 스토리

문서 본문은 4개의 Act로 구성된다.

1. **Act 1. 진행한 방향** — 무엇을 왜 어떻게 만들었나
2. **Act 2. 이슈 발생** — 실제 운영에서 드러난 문제들
3. **Act 3. 해결 방향** — 설계 결정과 대안 비교
4. **Act 4. 결과** — GenWorld를 통한 검증과 배운 점

### Layer 2 — Act 3 안의 각 설계 결정 = 미니 4박자

Act 3의 각 결정 항목(6개)이 다시 [진행한 방향 → 이슈 → 해결 → 결과] 4줄 패턴으로 풀린다. 이는 면접 시 "왜 그렇게 결정했어요?" 질문에 대한 직접적인 답변 재료가 된다.

## 4. 문서 내부 구조 (산출물 자체의 목차)

```
[메타블록]
- 프로젝트명 / 한 줄 정의 / 기간 / 역할 / 사용 기술 태그 / GitHub 링크

# Act 1. 진행한 방향
- 출발 동기와 초기 가설
- 첫 모양 (4 에이전트, 파일 기반 비동기, BOARD.md)
- 아키텍처 다이어그램 (Mermaid)
- [SCREENSHOT 슬롯] 4 에이전트 동시 동작 캡처

# Act 2. 이슈 발생
실제 git log·스펙 파일·커밋 메시지에서 발굴한 6개 이슈:
1. 라이브 시스템을 안전하게 멈출 방법이 없음 (수동 테스트 윈도우 확보 불가)
2. Windows에서 러너 스크립트 CWD 문제로 깨짐 (커밋 aad2d65)
3. Resume mode 파싱이 stdout에 잡음을 흘림 (커밋 2b253b6)
4. Loop runner가 조용히 죽음 (커밋 652d5f7)
5. 도큐먼트가 부족한 프로젝트에서 잘못된 결정
6. 운영자가 매번 어떤 .bat을 실행해야 할지 헷갈림

# Act 3. 해결 방향 (각 항목 미니 4박자)
3-1. FREEZE / DRAIN_FOR_TEST 제어 플래그              ← 이슈 #1 대응
3-2. Preflight 단계 + 템플릿 스캐폴드                 ← 이슈 #5 대응
3-3. 6단계 라이프사이클 모델                          ← 이슈 #6 + 전체 운영 모델 정리
       (Preflight→Bootstrap→Seed→Operate→Control→Test)
3-4. 단일 Entry Hub `orchestration-tools.bat`         ← 이슈 #6 대응
3-5. Windows 런타임 안정화 패치 묶음                  ← 이슈 #2/#3/#4 동시 대응
       (CWD 수정, 파싱 출력 정리, ASCII-safe 설정 키)
3-6. 9개 언어 README                                  ← (이슈 대응이 아닌 확산 전략, 별도 동기)

주의: 이슈와 해결안은 엄격한 1:1이 아니다.
- 해결안 3-5는 다중 이슈를 한 번에 묶는 안정화 패치 묶음
- 해결안 3-3, 3-6은 특정 버그 대응이 아닌 시스템 설계/확산을 위한 능동적 결정
이 사실 자체가 시니어 시그널 — "버그를 고친 것"과 "구조를 만든 것"을 구분해서 풀어낸다.

# Act 4. 결과
- GenWorld 검증 데이터 (5일·1141 커밋·S-001~S-113+)
- 시스템 목록 (GameManager / CombatManager / SkillSystem / InventorySystem / QuestSystem / SaveSystem / AI·NPC dialogue)
- 최근 커밋 인용 5~6개
- 프레임워크 자체 재사용성 (Unity/Godot/Unreal 자동 감지)
- 배운 점 / 다음에 다르게 할 것 3~4개
- [SCREENSHOT 슬롯] GenWorld 인게임, BOARD.md 운영 중
```

## 5. 작성 시 지켜야 할 원칙

### 5.1. 모든 숫자는 검증 후 기재

- "1141 커밋", "5일", "113+ 태스크" 등 모든 정량 지표는 작성 시점에 다시 `git log`로 검증한다.
- 검증 불가능한 숫자는 적지 않는다.

### 5.2. 인용은 실제 git log·파일 내용에서

- 커밋 메시지를 인용할 때는 실제 커밋 해시와 함께 적는다.
- 코드 발췌 시 GitHub permalink (특정 커밋 SHA) 사용을 우선한다.

### 5.3. 설계 결정은 항상 "대안"을 함께 적는다

- "왜 파일 기반인가" → "vs. 인메모리 큐, vs. API 콜" 비교를 포함한다.
- 시니어/AI 엔지니어 신호의 핵심은 "다른 선택지를 알면서 이걸 골랐다"이다.

### 5.4. 미니 4박자는 4~6줄로 짧게

- Act 3의 각 항목은 짧게. 길어지면 본 줄기가 흐려진다.
- 추가 설명이 필요하면 부록 D 코드 큐레이션에 링크로 넘긴다.

### 5.5. 캡처는 슬롯만 표시, 실제 이미지는 사용자가 채움

- `[SCREENSHOT: 설명]` 형식의 플레이스홀더를 본문에 박아둔다.
- 어떤 캡처가 필요한지 부록 F에 일람으로 정리한다.

### 5.6. 본문 외 부록은 발췌용 재료 저장소

- Act 본문은 "스토리"
- 부록 A~F는 "이력서·포트폴리오에서 잘라 쓸 수 있는 토막"
- 두 영역의 정보가 일부 중복되어도 문제 없다 (의도된 중복).

## 6. 작업 절차

1. 이 디자인 명세 작성 및 자체 리뷰 완료 (현재 단계)
2. 사용자 검토 → OK 시 다음 단계
3. `writing-plans` 스킬로 산출물 작성 절차를 단계별 플랜으로 분해
4. 플랜에 따라 산출물 마크다운 파일을 작성
5. 작성 중 모든 정량 데이터 재검증 (git log, 파일 카운트)
6. 작성 완료 후 사용자가 검토하고 발췌해서 기존 PDF 포트폴리오·이력서에 반영

## 7. 산출물 검증 기준

산출물 파일이 다음을 모두 만족하면 완료로 본다.

- [ ] 4 Act 본문이 모두 작성되어 있음
- [ ] Act 3의 6개 결정 항목이 각각 미니 4박자로 작성됨
- [ ] 부록 A~F가 모두 채워져 있음 (단, 사용자가 직접 채울 캡처는 슬롯만)
- [ ] 본문에 등장한 모든 숫자가 git log로 재검증됨
- [ ] 본문에 등장한 모든 커밋 해시가 실재함
- [ ] Mermaid 다이어그램이 1개 이상 포함됨
- [ ] GitHub 링크가 부록 D·E에 모두 작동하는 형태로 존재함 (public 레포 기준)
- [ ] 이력서 bullet이 7~10개 포함되어 있음

## 8. 명시적으로 하지 않는 것 (Out of Scope)

- 새로운 포트폴리오 사이트 디자인
- PDF 레이아웃 디자인
- 영문판 작성 (한국 시장 타겟이므로)
- 면접 예상 질문집 (사용자가 별도 생성 의사 없음)
- 코드 변경 (이 작업은 문서 작업이며 프레임워크 코드는 건드리지 않음)
